# Provenance Guard

A backend attribution system for creative-sharing platforms. It classifies whether
submitted text was likely written by a human or generated by AI, scores confidence
in that verdict, surfaces a plain-language transparency label, and lets creators
appeal a classification.

\---

## Architecture Overview

A submission's raw text is scored independently by two signals — an LLM classifier
(semantic) and stylometric heuristics (structural, including a burstiness sub-metric).
Their scores are combined into a single confidence value, which maps to one of three
transparency labels. The full decision (both signal scores, the combined confidence,
and the chosen label) is written to a structured SQLite audit log and returned to the
client with a unique `content\_id`. An appeal looks the content up by that
`content\_id`, sets its status to `under\_review`, appends the appeal to the audit log
beside the original decision, and returns a confirmation.

*(See `planning.md` for the architecture diagram of both flows.)*

## Endpoints

|Method|Route|Purpose|
|-|-|-|
|POST|`/submit`|Classify text; returns `content\_id`, confidence, label|
|POST|`/appeal`|Contest a classification by `content\_id`|

## Detection Signals

The system uses \*\*two distinct signals\*\* — one semantic, one structural — chosen so

their blind spots don't overlap.



\*\*Signal 1 — LLM Classification (Groq, semantic).\*\* Captures a holistic judgment of

whether the text \_reads\_ as human: voice, coherence, tonal tells. Logged as

`llm\_score` (0-1, higher = more AI-like). Chosen because it sees things no formula

can. \*\*Misses:\*\* it's a non-deterministic black box and can be fooled by

lightly-edited AI text.



\*\*Signal 2 — Stylometric Heuristics (structural).\*\* Measures statistical uniformity

via type-token ratio (unique words ÷ total words) and sentence-length variance

(burstiness). These are combined into a single `stylometric\_score` (0-1, higher = more AI-like). The `burstiness\_score` is logged separately for debugging and transparency, but it feeds into the stylometric score rather than being used as an independent signal. Chosen because AI text is measurably more uniform than human writing, and this is fully independent of the semantic signal. \*\*Misses:\*\* confuses "formal and even" with "AI," so it can false-flag polished or non-native human writing.



The two are paired \_because\_ their blind spots don't overlap: when they agree,

confidence is high; when they \*\*disagree\*\*, that disagreement drives the "uncertain"

label rather than a forced guess (see the bob example below).



## Confidence Scoring

\## Confidence Scoring



The system uses \*\*ensemble detection\*\* with three signals combined into a single confidence score in `\[0, 1]` (higher = more likely AI), which is then mapped to three bands. The LLM signal reflects semantic judgment; the stylometric signal anchors the verdict in measurable text properties; and the burstiness signal captures lexical variation patterns.



\*\*Exact combining formula from the code:\*\*



```python

def combine\_signals(llm\_score, stylometric\_score, burstiness\_score=None):

&#x20;   """

&#x20;   Combine detection signals into a single confidence score.

&#x20;   With ensemble: 45% LLM, 30% Stylometric, 25% Burstiness

&#x20;   Without ensemble: 60% LLM, 40% Stylometric

&#x20;   """

&#x20;   if burstiness\_score is not None:

&#x20;       # Ensemble mode with conflict resolution

&#x20;       scores = \[llm\_score, stylometric\_score, burstiness\_score]

&#x20;       weights = \[0.45, 0.30, 0.25]

&#x20;       

&#x20;       # Check for signal conflicts (disagreement > 0.3)

&#x20;       max\_diff = max(scores) - min(scores)

&#x20;       if max\_diff > 0.3:

&#x20;           # High conflict - reduce confidence toward uncertain

&#x20;           weighted\_score = sum(s \* w for s, w in zip(scores, weights))

&#x20;           # Pull toward 0.5 (uncertain) proportionally to conflict

&#x20;           conflict\_factor = min(1.0, (max\_diff - 0.3) / 0.4)

&#x20;           adjusted\_score = weighted\_score \* (1 - conflict\_factor) + 0.5 \* conflict\_factor

&#x20;           return adjusted\_score

&#x20;       

&#x20;       return sum(s \* w for s, w in zip(scores, weights))

&#x20;   else:

&#x20;       # Standard two-signal mode

&#x20;       return (llm\_score \* 0.6) + (stylometric\_score \* 0.4)



**Validation - two real submissions with clearly different scores** (from the audit
log):

* **High-confidence human** — input: *"I scribbled this on the back of a receipt while
waiting for the bus. My handwriting was awful. The coffee had gone cold hours ago,
but I drank it anyway because I needed something to do with my hands."* -> confidence
**0.144**, label **likely\_human** (`llm\_score` 0.2, `stylometric\_score` 0.18). Both
signals agree the text reads as human — rich, varied sentence structure and a
personal voice — so confidence lands well inside the human band.
* **Lower confidence** — input: *"In today's rapidly evolving digital landscape, it is
important to note that effective communication plays a crucial role. Furthermore,
leveraging synergies can optimize outcomes..."* -> confidence **0.5**, label
**uncertain** (`llm\_score` 0.8, `stylometric\_score` 0.17). Here the two signals
**disagree sharply**: the LLM reads the buzzword density as AI-like while stylometry
reads it as human. The system does not force a call - the disagreement resolves to
the uncertain band, which is the correct, honest outcome.

The stark contrast between these scores validates the combined-signal approach:
agreement between signals yields a confident verdict, while genuine disagreement is
surfaced as uncertainty rather than hidden behind a binary flip.

For a real deployment I would implement a more sophisticated calibration process using
a large, diverse labelled dataset rather than heuristic anchor constants, and I would
investigate the burstiness metric (which reads 0.0 on most current samples). I might
also add further independent signals (e.g. sentiment or named-entity patterns) to
strengthen the ensemble.

## Transparency Label

The label shown to readers changes with the confidence band. Exact text of each
variant, as emitted by the system:

|Result|Label text (verbatim)|
|-|-|
|Likely HUMAN|`"Our analysis suggests this content was likely written by a human. The writing shows natural variation and patterns consistent with human authorship. No further verification is needed at this time."`|
|UNCERTAIN|`"We're unable to confidently determine whether this content was written by a human or generated by AI. Some patterns are ambiguous, which is common with certain writing styles. The creator can provide additional context to help clarify."`|
|Likely AI|**Our analysis suggests this content may have been AI-generated. While our detection systems show a high degree of confidence ,automated tools can make mistakes. If you're the creator and believe this is an error, you can appeal this classification.**|

A verified-human creator (see stretch features) receives a `✅ Verified Human Creator | ` prefix on the human label — this prefix appears only when `verified\_human = 1`; the plain human text above is the default. The image-description content type emits the same uncertain text with "content" replaced by "image description".

## Appeals Workflow

Creators contest a result by POSTing `content\_id` and `creator\_reasoning` to
`/appeal`. The system sets the content's status to `under\_review`, logs the appeal
next to the original decision, and confirms receipt. No automated re-classification.

Real example from the logs: creator `bob` appealed content
`3c2eb313-4564-4ccc-b3a6-f8ed3acbc26f` with the reasoning *"I wrote this in a formal
business tone for a corporate report. It is stiff but it is genuinely mine."* The
entry's status flipped to `under\_review` and the reasoning was stored alongside the
original classification.

```bash
curl -s -X POST http://localhost:5000/appeal \\
  -H "Content-Type: application/json" \\
  -d '{"content\_id": "3c2eb313-4564-4ccc-b3a6-f8ed3acbc26f", "creator\_reasoning": "I wrote this in a formal business tone for a corporate report. It is stiff but it is genuinely mine."}'
```

## Rate Limiting



Applied to `/submit` via Flask-Limiter (`memory://` storage). Chosen limits:

\*\*`10 per minute; 100 per day`\*\* per client IP.



Reasoning: a real writer polishing and re-submitting their own work rarely exceeds a handful of submissions per minute, so 10/min leaves comfortable headroom while a flooding script trips it on the 11th request; the 100/day cap bounds sustained abuse from a single source without affecting any plausible individual creator.

## Known Limitations

**Formal / templated human writing lands in "uncertain."** bob's real submission
(confidence 0.5) is the clearest case: genuinely human corporate prose is uniform
enough that stylometry reads it as human (0.17) while the LLM reads the buzzword
density as AI (0.8). The signals disagree, so the system correctly declines to call
it - but this means a whole class of legitimate formal writing will never get a
confident "human" verdict. This is the false-positive-avoidance tradeoff working as
designed, but it's a real limitation for business/academic content.

**The burstiness metric currently reads 0.0 on most samples**, including
multi-sentence texts, so at present it contributes little to the stylometric signal.
Until that's investigated, the structural signal leans almost entirely on the other
sub-metric (type-token ratio), which weakens it on short inputs where that is already
noisy.

the interaction between the semantic and structural detection signals. It helped me think

through \*why\* they should be combined and how their disagreement should drive the "uncertain"

## AI Usage



1\. \*\*Prompt for Scoring Function:\*\* I directed the AI to generate the core confidence scoring function based on the two signal values from `planning.md`. The output was a function that simply returned the average of the two scores. I had to revise this to implement the 50/30 weighting with the 0.1 constant term, and the logic for the three classification bands, ensuring it matched the requirements in `planning.md` §2.



2\. \*\*Prompt for Stylometric Helpers:\*\* I asked the AI to write the helper functions for the stylometric heuristics (`sentence\_variance` and `type\_token\_ratio`). It produced a basic implementation that did not handle edge cases like empty strings or texts with no sentences. I had to override this by adding robust error handling and fallback logic to ensure the system wouldn't crash on malformed input or very short texts. I also had to debug why `burstiness\_score` was consistently returning 0.0 — the AI's initial implementation didn't handle the variance calculation correctly when sentence lengths were identical.

## Stretch Features

**\*\*Multi-modal support.\*\* The pipeline accepts a `content\_type` of `image\_description` in addition to `text`. When `content\_type` is `image\_description`, the system runs the same LLM and stylometric signals on the description text, but the transparency label is modified slightly to reference "image description" instead of "content". This is visible in carol's audit log entries, where the label reads: \*"We're unable to confidently determine whether this image description was written by a human or generated by AI."\***



**\*\*Verified-human credential.\*\* Content from a verified creator carries a `verified\_human` flag (1 = verified). When this flag is set, the human transparency label is prefixed with `"✅ Verified Human Creator | "` to distinguish verified accounts from the base classification. This is visible in alice's entries where the label begins with the verified prefix. The verification step is a simple boolean flag set on the creator account — in a production system this would involve a more robust verification process, but for this implementation it demonstrates the concept of a provenance certificate.**
